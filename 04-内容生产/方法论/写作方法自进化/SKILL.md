# 写作方法自进化

> 版本：v1.1 · 2026-06-15
> v1.0→v1.1：**路径修正**。步骤 1 统一使用 `05-项目运营/`。

## 为什么需要这个 Skill

不管方法论写得多细，AI 写出来的东西总和用户自己写的有差距。最有价值的训练数据是「AI 原稿 → 用户改完的终稿」这个 diff——用户每次改 AI 稿的动作里包含了他的判断标准和偏好，比任何方法论文档都精确。如果不把每次改稿沉淀为规律，改完就丢了。

核心做法：把「用户如何修改 AI 原稿」转成可复核的数据，再用多份样本提炼稳定规律。

## 触发条件

- 用户说「写作方法自进化」「自我进化」「学我的写作方式」「学我的修改方式」「积累修改规律」
- 用户说「每次写的东西都要我改几遍」「自动学我的偏好」
- 周复盘时顺带检查是否有至少 3 对新的可对比文章
- 改完 3-5 篇后就值得跑一轮

## 前置条件

每个高频内容账号的目录结构：

```
05-项目运营/{项目}/
├── 待审核/    ← AI 写完先放这里（原稿不动，是训练集）
└── 已发布/    ← 用户改完发的版本放这里
```

两份同名文件就是一对训练样本。至少凑齐 3 对才执行——少于 3 对找不出共性规律。

### 备选路径：对话纠正提取

当文章 diff 样本不足（< 3 对同名文件）时，**从对话存档直接提取用户对 AI 的纠正规律**：

1. 扫描最近 10-14 天的对话存档（`00-运转日志/对话记录/`）
2. 搜索纠正信号词（纠正/不对/不是这样/改成/应该是/误判/教训）
3. 提取跨至少 2 次对话出现的共性纠正模式
4. 产出写入 `01-关于我/AI协作偏好.md`（非文章写作的修改偏好）或 `04-内容生产/方法论/{对应子类}/修改规律.md`（文章写作偏好）
5. 每条规律标注来源对话文件（`[[双向链接]]`）

此路径覆盖用户说「每次写的东西都要我改几遍」「自动学我的偏好」但尚无结构化文章对的场景。

## 执行流程

### 步骤 1：扫描同名对

扫描 `05-项目运营/` 下各项目的 `待审核/` 和 `已发布/` 目录，找出文件名相同（或高度相似）的文件对。

### 步骤 2：门控检查

同名对少于 3 对 → 报告"样本不足，当前 N 对，需要至少 3 对"并退出。≥3 对 → 继续。

### 步骤 3：逐对 diff

逐对对比 AI 原稿和用户终稿，标注每处修改的类型：

- **删**：用户删掉的内容
- **替**：用户替换的表述
- **加**：用户新增的内容
- **重排**：用户调整的结构/顺序
- **改语气**：用户调整的语气/态度

### 步骤 4：跨对找共性

跨至少 2 对出现的相同修改模式 → 立为规律。单次出现的修改不立规律（统计学不允许从 n=1 推断）。

### 步骤 5：写入修改规律

规律按账号类型归类，写入 `04-内容生产/方法论/{对应子类}/修改规律.md`。子类对应关系：内容创作 / 营销销售 / 搜索 SEO 等，根据项目所属账号判断。

### 步骤 6：下次生效

写入后，AI 下次为该账号写作时自动读取对应 `修改规律.md` 并遵守。

## 升级机制

某条规律在 **3 轮进化里反复出现** → 状态从 `观察中` 升级为 `已固化` → 提请用户确认是否将该规律从「修改规律」升级到「方法论正文」。

## 输出格式

写入的每条规律使用以下格式：

```markdown
### 规律：[一句话描述]
- **状态**：观察中 / 已固化
- **出现次数**：N/3 轮
- **修改类型**：删/替/加/重排/改语气
- **典型案例**：原文 → 改后
- **适用场景**：[什么时候用这条规律]
```

## 约束

- **不自动修改方法论正文**：只能写到「修改规律」单独区域，升级到正文须经用户确认
- **不删除原稿**：`待审核/` 里的 AI 原稿是训练集，永远不删
- **不从单次修改总结规律**：至少 2 对重复才立规律
- **本 Skill 是跨所有写作子类的元机制**：产出的修改规律按账号类型分流到对应子方法论下

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 待审核目录全空 | 所有项目的 `待审核/` 都没文件 | 不报错退出，告知「尚无 AI 原稿可对比。从现在开始每次 AI 写稿先放待审核/，积累 3 对后可执行」→ 走备选路径：从对话存档提取纠正规律 |
| 有文件但不足 3 对 | 同名对 < 3 | 告知当前对数，走备选路径（对话纠正提取），不强行对比 |
| 已发布缺对应原稿 | `已发布/` 有文件但 `待审核/` 找不到同名 | 记录为「单向样本」，不计入可对比对，跳过不报错 |
| 对话存档中无纠正信号 | 备选路径扫描无结果 | 告知「近期无可提取的纠正规律」，不编造规律 |
| 跨对找不到共性 | 3+ 对 diff 完成但无 ≥2 次重复的修改模式 | 报告「本次对比未发现跨对共性规律，修改模式为单次偶发」，不入库 |
| 同名文件内容差异过大 | 同名文件相似度 < 30%（可能放错了文件） | 提醒用户确认是否为同一篇文章，不确定则跳过该对 |
| 路径引用的目录不存在 | `05-项目运营/` 下找不到对应项目 | 列出实际存在的项目目录，等用户确认路径 |

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **不足 3 对强行对比** | 1-2 对找不出统计学有效的共性规律，立规律是自欺欺人 | 走备选路径（对话纠正提取），等文章对自然积累 |
| 2 | **从单次修改推断规律** | n=1 可能是用户随手改的，不是系统性偏好 | 跨至少 2 对重复出现才立规律 |
| 3 | **在没有新文章时假装跑过** | 周复盘触发检查，没新数据就说没新数据 | 如实报告「本次检查：无新可对比对」 |
| 4 | **直接改方法论正文** | 修改规律未经用户确认就写进正文，可能引入错误标准 | 规律先写「修改规律」区域，升级需用户显式确认 |
| 5 | **删除待审核原稿** | 原稿是训练集，删了就没法对比 | 永远不删待审核文件 |
| 6 | **不区分改稿类型就立规律** | 同一删改操作在不同账号/文体下含义完全不同 | 规律按账号类型分流写入对应子目录 |
