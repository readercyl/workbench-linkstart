# 全系统诊断

> 版本：v1.1 · 2026-07-14 · 加结果汇聚
> v1.0→v1.1：① 汇总升级为四维度表（不只一句话）；② 跨层交接：运营层修复后→数据层复扫；③ 12步→10步修正

## 定位

编排层，不包含诊断逻辑。按三层顺序调度，汇聚结果出汇总报告。

## 流程

### ① 运营层巡检

调用巡视模式，产出 `.sh-state.md`。有修复 → 执行修复后再跑一次轻扫确认。

### ② 数据层体检

调用快速模式（S1）。运营层有修复 → 必须跑数据层确认知识管线未受影响。全通过→静默，失败→标注。

### ③ 框架层审核

调用全量 10 步审计。产出一份报告包含：发现明细（🔴🟡）+ 优化建议（🔵）+ Token 审计。

### ④ 汇总

三层各自跑完后，汇聚输出：

```
🏥 全系统诊断 · HH:MM

| 层 | 状态 | 异常 | 处置 |
|----|:----:|------|------|
| 运营层 | 🟢/🔴 | N 项 | 修复 M 项 |
| 数据层 | 🟢/🟡 | N 项 | 标注 |
| 框架层 | 🟢/🟡 | N 项 | 修复 M 项，优化 K 项 |

⚠️ 跨层关注：[运营修复影响数据层？/框架改动需验证？]
```

## 铁律

- **不替代三层**：入口而已，用户仍可直接调任一子 Skill
- **运营修复后复扫数据层**：运营层动了脚本/路径/知识管线→数据层必须确认
- **不静默**：必定输出汇总
