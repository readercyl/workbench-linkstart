# 工具管理

> 版本：v1.4 · 2026-07-14 · 为什么需要并入角色

> 统一管理本地所有工具：Skill、MCP、CLI。解决「装了什么、装在哪、有没有重复、需不需要更新」四个问题。

## 角色

维护系统框架层——工具注册表。清点/安装/体检/评估四合一。工具散落在 7+ 个目录加 MCP/npm/pip，没有人能记清全貌。被工具自修正和框架层审核引用。

## 模块二：安装与归位

### 触发

用户说：「装一下 XX」「安装这个 skill」「https://github.com/xxx 帮我装」

### 安装流程

1. **获取**：git clone 或 npx/npm/pip 安装
2. **判断归属**：

| 工具功能 | 归属目录 |
|---------|---------|
| 内容生产相关（写作/排版/发布/配图） | `04-内容生产/skills/` |
| 对外分享/展示相关（PPT/小说/演示） | `06-对外分享/skills/` |
| 认知学习相关（教学/诊断/思维） | `03-认知工坊/skills/` |
| AI 产品开发相关（编程/部署/测试） | `07-AI产品开发/skills/` |
| 通用/跨领域 | `~/AI全局技能包/` |

3. **判定归属目录（领域/中立/不可移动）并完成归位**：先判断安装方式是否允许移动本体。能移动 → 真实文件放对应领域目录或 `~/AI全局技能包/{name}/`，软链接从外面指进来。npm/pip/pipx 等固定安装路径的 → 不动本体，记入「全局安装不可移动工具」。同时检查是否重复安装——同一 Skill 在多个路径存在时，比较版本和内容，保留较新的、删除旧的。归属判断模糊时列出候选目录和依据，默认归到最相关的那个
4. **软链接**：`~/.claude/skills/{name}` → 实际路径
5. **同步三份清单**（强制执行，安装未完成前不得跳过）：
   - `工具清单.md`：同铁律「清单与分布联动」
   - `物理位置分布.md`：同铁律「清单与分布联动」
   - `CLAUDE.md` 工具索引：对应分组表新增一行（分组数 +1）
   - 如果新工具带来了新流程规则或注意事项，同步写入 Memory

### 位置调整

如果用户后续调整目录结构，只需：
- 移动实际文件夹
- 更新软链接 `ln -sf`
- 更新清单
- 历史日志不动

## 模块二·五：卸载与清理

触发：用户说「卸载 XX」「删掉 XX 工具」「不用 XX 了」。

流程：
1. **定位**：在工具清单中确认该工具的存在位置（Skill/MCP/CLI/脚本）
2. **移除软链接**：`.claude/skills/` 和 `~/.claude/skills/` 下的对应软链接
3. **删除源文件**（确认后）：`AI协作工具/skills/` 下或全局技能包下的本体
4. **清理 MCP 配置**：`~/.claude/mcp.json` 或 `settings.json` 中对该工具的引用
5. **检查依赖**：是否有其他工具/脚本/配置引用了该工具 → 列出受影响项，逐一更新
6. **更新清单**：工具清单 + 物理位置分布 + CLAUDE.md 工具索引同步删除条目

铁律：卸载前必须确认——不擅自删工具。确认后一次性清理干净，不留残留引用。

## 模块三：定期体检与维护

### 触发

用户说：「工具体检」「检查工具」「看看工具有没有问题」。周复盘时 AI 可提议工具体检。

### 体检项目

1. **断链检测**：`~/.claude/skills/` 下的软链接是否指向不存在的目标
2. **重复检测**：同一 Skill 是否在多个路径存在
3. **过期检测**：git clone 的 Skill，检查 `git fetch` + `git log origin/main..HEAD` 看上游有无新 commit
4. **中文缺失**：扫所有 SKILL.md 的 `description` 字段，标记中文占比 < 30% 的
5. **清单同步**：对比实际文件系统和清单，找出遗漏或多余的条目
6. **脚本可执行性**：检查 `AI协作工具/skills/*/` 下所有 `.sh` 文件是否有执行权限、是否 bash 语法正确（`bash -n`）

### 更新检查

对有 git 远端的 Skill：
```bash
cd {skill_path} && git fetch --quiet 2>/dev/null && git rev-list --count HEAD..origin/main 2>/dev/null
```
输出 > 0 表示有更新。生成更新提醒列表，等用户决定是否 `git pull`。

## 模块四：安装评估与整合

### 触发

用户给一个 GitHub URL 或工具名，说「这个怎么样」「要不要装」「帮我看看这个」。**这是前置步骤——装之前必须先评估。**

### 评估流程

1. **扫现有库**：调模块一，拿到当前全部主入口清单
2. **读 GitHub**：用 WebFetch 获取该工具的 README/SKILL.md，提取：功能描述、技术栈、依赖、与已有工具的相似度
3. **对比分析**：逐条对比现有工具，判断：
   - **冲突**：功能高度重叠 → 列出重叠点，建议选一
   - **互补**：功能不重叠但可串联 → 给整合建议
   - **新领域**：现有工具都没覆盖 → 标注为「补缺」
4. **结合实际**：基于用户当前的工作重心（从 `01-关于我/人物档案.md` 的「当前在做」和知识库的调用热点推断），判断这个工具的实用性。不装「以后可能用」的工具
5. **输出建议**：控制在 5-8 行内，包含：一句话定论、冲突/互补分析、建议（装/不装/替代方案）

### 整合方案

当发现互补工具时，主动探讨整合可能：

**整合信号**：
- A 的输出是 B 的输入（如 guizang → 公众号发布）
- A 和 B 覆盖同一流程的不同阶段（如 taste-skill 做设计 + impeccable 做审计）
- 多个工具共享同一套配置/资源

**整合形式**：
- 轻度：写一个编排 Skill，串联调用顺序
- 中度：共享配置文件和输出目录
- 重度：合并为一个 Skill，原工具作为子模块

整合方案直接展示给用户，等确认后执行。

## 铁律

- **先评估再安装**：不给 GitHub URL 直接装，必须先跑模块四
- **清单即事实**：每次操作后更新清单，清单与实际文件系统必须一致
- **归位不散落**：新工具必须放到正确的归属目录，不放临时位置
- **全量折叠，部分拆开**：父包全部子工具属于同一小组则折叠；只有部分相关则拆出单独列，来源标注所属父包
- **中文优先**：安装后立即检查并补中文描述，不等到体检才发现
- **不装未来工具**：「以后可能用」不构成安装理由。只装当前实际需要的
- **子工具只留主入口软链接**：baoyu/dbs/taste-skill 等系列只保留主入口的软链接（如 `baoyu-skills`、`dbs`、`taste-skill`），子工具不配独立软链接。AI 调主入口后按 SKILL.md 内部路由到子目录读取子工具文件。子工具文件本体不动，只是不往 system prompt 里塞
- **清单与分布联动**：工具清单.md 和物理位置分布.md 同步维护。安装/卸载/移动任一操作后两份文档同时更新，名称集合保持严格一致
- **本地优先于远端**：判断工具是否存在、是否需要删除——以本地实际文件和工具清单为准，不以 `.git` 目录里的 remote 地址为准。工具已脱离原始仓库独立运行，不再隶属于原项目
- **脚本也是工具**：`AI协作工具/skills/*/` 下的 `.sh` 脚本纳入清单和体检。新脚本创建后同步更新清单

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 安装失败 | `npx skills add` / `git clone` 报错 | 告知错误信息，回退：删除已创建的空目录，不留下半成品 |
| git fetch 超时 | 体检更新检查中远端不可达 | 标注「远端不可达」而不是报错退出，跳过该工具继续下一个 |
| 软链接目标不存在 | 断链检测发现 | 列出断链路径，提议删除或修复，不等用户确认不自动清理 |
| 清单与实际不一致 | 对比发现遗漏/多余条目 | 用实际文件系统为准更新清单，告知差异项 |
