# 知识编译

> 版本：v8.8 · 2026-07-14 · 深度整理判断优化 + MOC归入框架
> v8.7→v8.8：① 步骤 1 深度整理条件细化（场景+45分钟+万字+原文占比），Skill 名「对话深度整理」→「深度整理」② 步骤 3 子代理加「严禁提前过滤」③ 步骤 4 明确 grep 是定位工具不是判断工具 ④ 步骤 6 MOC 归入加两步锚定法（调用者+最近邻卡）

## 角色

维护系统数据层——外部知识→原子卡片的核心管线。上游接收微信清洗和深度整理的预处理产物，下游交付知识检索和数据层体检。**编译即归入——提取、写卡、MOC归入、校验同一次完成。只编译用户指定的文件。**

```
深度判断 → 通读列候选 → 对抗式查漏+过滤 → 语义检索去重 → 写卡 → MOC归入 → 同步记录 → 校验 → 归档 → 报告
```

---

## 流程

### 1. 深度整理判断

> 判断素材是否需要先走「深度整理」Skill 做结构化预处理，还是可以直接提取卡片。

**触发「深度整理」Skill 的条件**（满足任一）：

| 维度 | 阈值 |
|------|------|
| 场景 | 多人线下对谈、课程演讲、会议讨论等对话形态 |
| 录音时长 | >45 分钟 |
| 逐字稿字数 | >10,000 字 |
| 结构辅助判断 | 原文逐字稿占比 >50%——即使不满足上述阈值，大头在原文转写就要整 |

**不触发**（跳过深度整理，直接卡片化）：整体为结构化长文、信息密度极高、非对话形态的内容。

**硬门控**：必须显式输出 `深度判断：{文件名} → {需要/跳过}深度整理，原因：{触发/不触发的具体条件}`。未输出此行 → 流程未启动，不得进入步骤 2。

### 2. 通读 + 列初筛候选

逐篇全文通读，拉出所有独立信息点，形成候选标题列表（此时不写卡）。一个信息点一行。

⚠️ **此阶段不做任何过滤判断**——即使是明显重复或无价值的候选也应列入候选列表。过滤在步骤 3（子代理对抗式过滤）和步骤 4（主代理语义检索去重）中分步执行。

**硬门控**：必须逐条列出全部候选标题。格式：`候选列表（N个）：1. 标题一 2. 标题二 ...`。未输出此列表 → 第一遍提取未完成，不得进入步骤 3。

### 3. 对抗式查漏+过滤

> 子代理同时做两件事：查漏（原第二遍重读）+ 过滤（条件②③）。查漏和过滤合一，一次子代理调用完成，节省一次独立调用的 ~50 秒开销。

Spawn 独立子代理，角色定位：

> 「你是挑剔编辑+过滤者。主 Agent 已从源文件提取了 N 个初筛候选。你的任务分两步：
>
> **第一步：查漏。** 通读原文，找出主 Agent 漏掉的独立信息点。唯一标准：它是不是一个独立的、可被复用的信息点？如果是，列出来。列不出来就说明你认真找了但确实没有。
>
> **第二步：过滤。** 对查漏后的全部候选（主 Agent 的 N 条 + 你自己补充的 X 条），逐条检查两个条件，任一不通过 → 建议跳过：
>
> - **条件② 决策可调用**：6 个月后，用户在什么具体场景下会需要调用这条信息辅助决策？答不上来 → 跳过。参考框架——用户常做的决策场景包括：产品方向判断、定价与商业模式选择、内容策略与渠道选择、学习方法与工具选型、团队协作方式。
> - **条件③ 信息有价值**：是用户可用的判断依据/概念/案例，而非已被 Skill/脚本/宪法内化的 AI 运维经验。AI 运维经验的正例（不入卡）：编译流程的分批策略、卡片命名规范、门控机制设计。AI 运维经验的反例——看起来像操作细节但**是**用户判断依据的（应入卡）：创业中要先验证需求再开发产品、定价要基于用户需求价值而非成本。
>
> **约束**：
> - **严禁在查漏阶段做任何过滤判断**。查漏的唯一标准是「是不是独立信息点」，不考虑它有没有价值、能不能通过条件②③。过滤是查漏完成之后的事——顺序不能乱，提前过滤会污染查漏的完整性
> - 对自己新补充的候选与主 Agent 已有候选，使用**完全相同的过滤标准**。如果某个你自己补充的候选被自己的过滤条件拒绝——如实记录，这说明它本就不该进入知识库
> - 每条建议跳过的候选必须附**具体原因**，不能笼统写「答不上来」。格式：`跳过：{候选标题} → 原因：{在什么场景下不具备决策价值 / 已被哪个 Skill/脚本/宪法内化}`
> - 如果对某条候选是否应跳过不确定，标记为 `⚡ 不确定，交主 Agent 复核`」

子代理的输出即对抗式查漏+过滤证据。

**过滤三条件最终分工**：

| 条件 | 由谁执行 | 在哪步 | 工具 |
|------|:---:|------|------|
| ① 语义去重 | 主代理 | 步骤 4 | grep + mcp recall |
| ② 决策可调用 | 子代理 | 步骤 3 | 用户视角主观判断 |
| ③ 信息有价值 | 子代理 | 步骤 3 | 区分「用户判断依据」vs「AI运维经验」 |

**硬门控**：子代理返回后，主 Agent 必须输出 `对抗式：[文件名] → 子代理补充 X 个候选，过滤跳过 Y 条，保留 Z 条：1. 候选A 2. 候选B ...`（附跳过列表和原因）。未输出此行 → 步骤 3 未执行，不得进入步骤 4。

**安全网——主 Agent 快速复核**：主 Agent 扫一遍子代理的跳过列表（只看标题和跳过原因）。发现明显误杀 → 捞回保留列表。捞回需记录原因。此步骤轻量，不重新执行语义检索。

### 4. 语义检索去重

> 步骤 3 子代理已完成条件②③过滤。步骤 4 的唯一职责是**条件①语义去重**——对步骤 3 保留的候选执行。⚠️ **grep 是定位工具，不是判断工具**——无论 grep 命中几条，最终所有候选都要经过知识检索的语义搜索路径（读卡片标题+正文、概念级比对），不能以 grep 结果直接做跳过/保留的决定。grep 零命中也必须走 mcp recall 检索。

**前置——grep 机械预筛**（从原步骤 2 并入）：从每个保留候选提取 3-5 个最独特的关键词，`grep -l "关键词" 02-知识卡片/知识卡片/*.md` 做批量预筛，记录命中数。

**执行方式**：grep 命中数 > 0 的候选走 AI 语义判断，grep 命中 = 0 的候选走 mcp recall 检索。对每个候选，抽取核心概念（3-5 字，不是关键词——是「它在讲什么」的概念），调用知识检索 Skill 的语义搜索路径，定位相关 MOC → 读 top 3 匹配卡片的标题+正文。

**三种结果**：

| 重叠程度 | 判定 | 操作 |
|---------|------|------|
| ≥80% 语义重叠 | 重复 | 跳过，不写。在编译报告中记录 `🔗 重复：{候选} → [[已有卡]]` |
| 补充关系（同概念新角度/新案例） | 补充 | 不建新卡。Edit 已有卡：正文末尾追加新段落，`**来源**` 追加 `、[[新源文件]]`，相关链接补新链。报告记录 `📎 补充：{候选} → [[已有卡]]` |
| <50% 重叠 | 全新 | 进入写卡流程 |

**硬门控**：检索完成后必须显式输出 `语义检索：{文件名} → 跳过 X 条、补充 Y 条、新建 Z 条`。未输出此行 → 语义检索未执行，不得进入写卡。此门控与步骤 3 的对抗式声明同级——两道声明都没出现之前，步骤 5（写卡）不能开始。

### 5. 编写卡片

> ⚠️ 前序检查：步骤 3 的 `对抗式：...` 声明和步骤 4 的 `语义检索：...` 声明都已输出 → 方可进入写卡。任一缺失 → 回退执行。

固定四区格式。写法规范详见 `references/卡片编写规范.md`。

**卡片 frontmatter 模板**：

```yaml
---
created: YYYY-MM-DD
source_type: flomo|视频日记|文章|分享|课程|讨论|AI对话|个人实践
status: seed
---
```

`status` 生命周期：`seed`（默认）→ `growing`（调用≥3）→ `evergreen`（调用≥10）→ `dormant`（长期未调用），由体检 Skill 自动升级。编译时统一写 `seed`。

**来源可多源**：同一概念出现在多个源文件中时，`**来源**` 节用 `、` 并列：`[[flomo-2024]]、[[flomo-2026]]`。语义检索判定为「补充关系」时，Edit 已有卡追加来源，不覆盖原有来源。

**写卡后立即自检**（逐项确认）：

1. frontmatter 三字段完整（created/source_type/status=seed）？缺 → 补
2. `**来源**` 节存在且指向素材箱实际 `.md` 文件名？缺 → 补；`[[wikilink]]` 目标在 vault 中不存在 → 去掉方括号改为纯文本引用（禁止指向 .pdf、已删除文件、错误路径）
3. 正文 ≥ 50 中文字符实质内容？不足 → 扩充
4. `**来源**` 与 `**相关链接**` 之间有空行分隔？缺 → 补空行
5. `**相关链接**` 至少链 1 张已有卡？无 → 补；每条链路有意义？弱链 → 换
6. 标题 8-25 字符、非模板化命名、非并列式？不符合 → 改

**写卡后立即反向链接对称化**：对新卡链接的每张已有卡，用 Edit 在对方 `**相关链接**` 节末尾追加 `- [[新卡名]]`。追加后用 `grep "\[\[新卡名\]\]" 目标卡.md` 验证——grep 命中为 0 则补写未成功。

### 6. MOC 归入

每张新卡写完立即归入对应 MOC。语义判断归属，用**两步锚定法**：

1. **调用者锚定**：用户在做什么类型的决策时需要这张卡？
   - 产品方向/商业化 → MOC-商业模式诊断方法、MOC-产品与服务
   - 内容策略/渠道选择 → MOC-内容创作能力、MOC-传播与获客
   - 学习方法/认知升级 → MOC-认知与心智
   - 销售/转化 → MOC-销售与转化
   - 项目验证/赛道选择 → MOC-项目验证与赛道、MOC-战略思维与创业
   - AI 协作/知识管理 → MOC-AI 协作方法论、MOC-知识管理元层
   - 工程/自动化 → MOC-AI 工程与自动化
   - 执行力/时间管理 → MOC-执行力与时间
2. **最近邻卡锚定**：语义上最接近的已有卡在哪个 MOC？新卡大概率同一归属。以此交叉验证步骤 1 的判断。

在 MOC 文件的合适子分类下追加 `- [[卡片名]]`。

- 新增 MOC 子分类可自行创建
- 不新增顶级 MOC（12 个 MOC 体系固定）
- 矛盾检测（≤2 张跳过）：grep 关键词预筛 → AI 语义判断。认知演进→`取代：[[旧卡]]`；场景差异→`<!-- 张力 -->`

### 7. 同步调用记录

```bash
bash AI协作工具/skills/数据层/知识编译/同步调用记录.sh
```

脚本对比全库卡片与调用记录。退出码 0=全覆盖，1=打印缺失项需补写。

### 8. 编译后校验

```bash
bash AI协作工具/skills/数据层/知识编译/编译收口.sh --force
```

脚本完成：数字同步（3 处 sed）→ log.md 追加 → 校验。退出码：0=跳过 / 1=失败 / 2=完成。

### 9. 归档原始文件

移动原始文件到素材箱子对应分类目录。此步必须在报告之前执行。

### 10. 报告

`📦 编译完成：候选 X 条 → 子代理过滤保留 Z 条 → 语义检索跳过 A 条, 补充 B 条 → 新建 C 张卡（MOC分布），校验✅/❌`

> 报告末尾追加 `🔄 已过滤：Y 条`——列出被步骤 3 子代理过滤跳过的候选标题和简短原因，以及主 Agent 是否捞回。用户可据此发现误杀模式，反向改进过滤标准。

## 自我进化

用户纠正编译遗漏/错链/格式 → 走工具自修正流程。同一模式被纠正 ≥3 次 → 向用户提议修正本 Skill。子代理过滤过严导致遗漏 ≥3 次 → 调整子代理提示词中的过滤标准或决策场景参考框架；子代理过滤过松导致冗余卡 ≥3 次 → 收紧反例清单。

## 铁律

- **只编译用户指定的文件**：不擅自动未指定的素材
- **编译即归入**：每张新卡写完立即 MOC 归入，不攒批
- **链接铁律**：`**相关链接**` 只写裸 `- [[卡片名]]`，至少链 1 张，语义质量优先
- **决策可调用优先于信息完整性**：知识库是决策辅助系统，不是资料库。价格、货源、八卦、纯数据点不写卡。每张卡必须能回答「什么时候会用到它」
- **过滤不跳过，遗漏不放行**：三条件（语义去重 + 决策可调用 + 信息有价值）逐条过。条件②③由子代理独立判断，主 Agent 不得代劳。子代理误杀可通过主 Agent 快速复核捞回
- **知识库辅助用户决策，AI运维经验不入卡**：卡片应覆盖用户做决策时需要的判断依据/概念/案例。已被 Skill/脚本/宪法内化的 AI 运维操作经验不写卡——应改进系统，不是填充知识库
- **来源节单向引用**：`**来源**` 只存在于卡片中，源材料文件不得引用卡片
- **卡片四区结构不可变**：frontmatter → 标题+正文 → 相关链接 → 来源
- **来源可多源**：同一概念被多个源文件讨论时，`**来源**` 用 `、` 分隔。补充已有卡时追加来源，不覆盖原有来源
- **调用记录全覆盖**：每张新卡在 `知识卡片调用记录.md` 中占一行，全库一一对应
- **文件名改动自动同步引用**：任何涉及 `素材箱/` 下源文件的重命名操作，必须先运行 `重命名源文件同步引用.sh` 扫描引用并同步，再执行重命名。不依赖用户手动使用——AI 自动执行

> 写法规范：`references/卡片编写规范.md` · 大批量策略：`references/大规模编译指南.md`
