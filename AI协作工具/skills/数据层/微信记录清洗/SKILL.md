# WeFlow 记录处理 Skill (双层高精过滤管道)

> 版本：v1.2 · 2026-07-14 · 「这是什么」压缩

## 角色

维护系统数据层——微信聊天记录的独立预处理。双层管道清洗后产出干净文本，作为知识编译的素材源。

## 做什么

双层管道：Layer 1 结构解析（去系统噪音+翻译微信ID）→ Layer 2 大模型语义过滤（按保留/删除标准去闲聊）。原地覆盖写入，自动 .bak 备份。产出干净文本作为知识编译素材源，编译前先走「对话深度整理」。

---

## 何时触发

当用户发出以下指令或提及相关词时：
- “帮我把刚导出的 WeFlow 记录清洗一下”
- “过滤微信群聊记录”
- “处理新微信记录并脱水”
- “对素材箱的聊天记录进行去噪”

## 用法说明

### 核心命令

使用 Skill 自带的管道启动器 `scripts/run_pipeline.sh` 进行一键串联运行：

```bash
bash scripts/run_pipeline.sh "<WeFlow 聊天记录 .txt 文件路径或目录路径>"
```

#### 运行示例

##### 1. 清洗单个文件
```bash
bash scripts/run_pipeline.sh "素材箱/_待处理/微信记录/群聊_xxx_全部时间.txt"
```

##### 2. 批量清洗目录下所有符合条件的群聊/私聊 `.txt` 记录 (推荐)
```bash
bash scripts/run_pipeline.sh "素材箱/_待处理/微信记录"
```

运行完成后，脚本会输出直观的降噪百分比汇总报告，并将原始记录原地覆盖，备份存储为 `.bak` 文件。

**清洗后固定步骤**：清洗产物写入 `素材箱/_待处理/微信记录/` 后，**立即提醒用户调用知识编译 Skill**——微信记录是对话类素材，编译前先走「对话深度整理」。这从异常处理提升为正常流程末尾步骤，不让清洗产物烂在待处理里。

## 异常处理

流程中每一步都有失败可能。以下预定义 fallback，保证不静默失败。

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 文件不存在 | 指定路径无 `.txt` 文件 | 告知用户文件不存在，列出 `素材箱/_待处理/` 下实际有的文件，等用户确认路径 |
| 文件非 WeFlow 格式 | 内容不含 `wxid_` 特征 | 告知用户「文件可能非 WeFlow 导出」，展示前 20 行让用户判断是否继续 |
| 微信ID翻译缺映射 | `wxid_xxx` 找不到真实姓名 | 保留 `wxid_xxx` 原文，在汇总报告中列出未翻译的 ID 清单而不是静默替换为空白 |
| DeepSeek API 不可用 | API 超时/限流/余额不足 | Layer 1 清洗照常执行 → 告知用户「Layer 2 大模型过滤跳过，仅完成结构清洗」→ 文件标注 `_L1only` |
| 清洗后文件为空 | 过滤后剩余 0 条消息 | **不执行原地覆盖**，告知用户「清洗后无可保留内容，原始文件保留在 .bak 中」，等用户决定 |
| 备份文件已存在 | 同名 `.bak` 已存在 | 追加时间戳后缀 `_HHMMSS.bak`，不覆盖已有备份 |
| Layer 2 过滤结果异常 | 过滤率 >99% 或 <50% | 告知用户过滤率异常，展示抽样 10 条让用户判断是否采用结果 |
| 脚本执行失败 | `run_pipeline.sh` 报错退出 | 告知用户错误信息，保留原始文件不做任何修改 |
| 清洗产物待编译 | 清洗完成后的 `.txt` 在 `_待处理/` 中 | 主动提醒用户可调用知识编译 Skill 编译。WeFlow 记录是对话类素材，编译前先调用「对话深度整理」Skill |

**🔴 CHECKPOINT**：以下操作前必须暂停等用户确认：
- 首次清洗不熟悉的群聊记录（确认清洗范围和保留标准）
- 清洗后文件为空时（确认是否放弃原始记录）
- 批量清洗超过 5 个文件时（确认批量范围）

**原则**：宁可不洗，不可洗坏。任何异常先告知，再按规则处理。绝不静默跳过。

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **跳过 .bak 备份直接覆盖** | 清洗结果不满意时无法恢复原始文件 | 永远先 `cp file.txt file.bak` 再操作 |
| 2 | **静默跳过 Layer 2 失败** | 大模型过滤是整个流程的核心价值，跳过不告知等于给用户一个不完整的结果 | 明确告知并标注文件 `_L1only` |
| 3 | **对已清洗过的文件再次清洗** | 二次清洗会把保留的高价值内容再次过滤，损失信息 | 检查文件是否有 `.bak` 备份或 `_cleaned` 标记，有则提醒用户 |
| 4 | **不读文件内容就开始清洗** | 不了解内容结构就直接跑脚本，可能把非 WeFlow 格式文件洗坏 | 先用 `head -50` 确认文件格式 |
| 5 | **假设备份一定成功** | 磁盘满或权限问题会导致备份静默失败，之后的覆盖不可逆 | 备份后 `ls -la` 确认 .bak 文件存在且大小 >0 |
| 6 | **把 .txt 以外格式的文件拉进管道** | 二进制/PDF/图片文件被当文本处理会损坏 | 先 `file` 命令确认文件类型，非 UTF-8 文本则拒绝处理 |
| 7 | **清洗完成后不报告降噪率** | 用户不知道清洗效果好坏，无法判断是否采纳结果 | 输出 `原始 X 条 → 清洗后 Y 条，降噪 Z%` |
