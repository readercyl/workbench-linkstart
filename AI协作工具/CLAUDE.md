# AI协作工具·子宪法

> 版本：v1.2 · 遵从根宪法 v2.0 · 2026-07-07

## 目录结构

```
AI协作工具/
├── skills/              ← Skill 定义的唯一存放处
│   ├── 运营层/           ← 对话存档 · 日报 · 周复盘 · 运营层巡检
│   ├── 数据层/           ← 编译 · 检索 · 体检（含链接校验+网络诊断子模块）· 深度整理 · 微信清洗
│   └── 框架层/           ← 框架层审核（含流程评估子模块+Token审计）· 工具管理 · 工具自修正
├── hooks/               ← 自动化钩子脚本（.sh 在 Obsidian 不可见，说明见目录内 CLAUDE.md）
├── MCP工具/              ← MCP Server 使用说明
├── CLI工具/              ← 本地 CLI 工具（weflow/wx-cli）
├── api配置/              ← API Key/认证配置参考
└── CLAUDE.md            ← 本文件
```

## 铁律

- **Skill 是唯一权威**：每个 Skill 的 SKILL.md 定义它的行为。工具清单、使用说明都在 Skill 自身目录内，不在顶层另建 `工具管理/` 之类的重复目录
- **不在此目录重复存放工具清单**：统一清单在 `skills/工具管理/工具清单.md`，由「工具管理」Skill 维护
- **不在此目录存放历史文档**：质量检测报告归 `00-运转日志/体检报告/`
- **skills-lock.json 在 ~/.claude/**：不在项目内。那是 Claude Code 全局配置
- **新 Skill 安装**：先走 skill-creator 判断 → 按功能归入对应板块的 `skills/`（03认知/04内容/06对外分享/07产品开发）→ 在 `.claude/skills/` 建软链接
- **目录改动后自检**：顶层新增/删除目录后，立即更新本文件的目录结构节
- **新工具安装后同步三份清单**：装完立即同步 `工具清单.md` + `物理位置分布.md` + `CLAUDE.md` 工具索引，不得等用户提醒

## 关键约束

- 创建新 Skill → 必须走 `skill-creator`，不手写 SKILL.md
- 修改已有 Skill → 可直接 Edit SKILL.md
- 结构性改动后 → 立即调「框架层审核」Skill 做运转层一致性检查
- Hook 脚本改动后必须测试
- 本目录的 Skill 定义对所有 AI 工具（Claude Code、Codex、Gemini CLI）生效
