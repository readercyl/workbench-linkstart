# Skill 自修正

> 版本：v3.0 · 2026-07-14 · 为什么需要+覆盖范围并入角色

## 角色

维护系统框架层——Skill 和脚本的自我修复机制。每次用户纠正 AI = 一次免费工具测试 → 根因分析 → 修正 SKILL.md/.sh → 验证。覆盖自建 Skill（`AI协作工具/skills/` 下 13 个），不管宪法/自动化链/配置（归框架层审核）。脚本只修确定性 bug，业务逻辑变更需用户确认。

## 触发条件

以下信号触发本 Skill。**触发信号出现时，AI 必须自动执行本流程（根因分析→提议修正），不需要等用户显式说「做自修正」。修完眼前的错误不算完——必须接着做工具修正。**

- **用户指出 AI 执行有误**：「不对」「你理解错了」「这个字段不是这么填的」
- **用户补充 AI 遗漏的规则**：「你忘了做 X」「这个步骤你跳过了」
- **用户否定 AI 的方案并给出正确做法**：「应该这样而不是那样」
- **用户要求 AI 检查并修正**：「看看有没有问题」「把这个写进去」
- **用户要求优化/审计 Skill**：「优化一下 skill」「审计 skill」「帮我看看这个 skill 有没有问题」「一对一优化」——触发主动审计模式（见下文「主动优化流程」）
- **用户要求回顾/纠错体系**：「做回顾和纠错」「梳理一下流程」「评估一下这个体系」——触发体系审计模式。与单个 Skill 优化的区别：范围是整个 Skill 链路或子系统，不是单个文件；需要逐环节对照 Skill 定义 vs 实际执行找差距；产出是系统性修正方案而非单点修

**不触发的**：
- 用户对内容本身的异议（如「这段话写得不好」→ 内容问题，不是工具问题）
- 闲聊中的建议（如「以后可以试试别的方法」→ 不是纠正）
- 与工具无关的操作纠正（如「这个文件路径错了」→ 单次操作错误，不是工具定义问题）

## 流程

### 第一步：根因分析

用户纠正发生时，立即判断错误根因在哪里：

| 根因 | 处理方式 |
|------|---------|
| 工具定义不清晰/有歧义 | → 第二步：修正 |
| 工具定义缺少边界条件 | → 第二步：补充规则 |
| 工具定义有规则但 AI 没遵守 | → 不改工具，但要自问：为什么没遵守？是规则太隐蔽还是自己太粗心？ |
| AI 自己的理解偏差，工具本身没问题 | → 不改工具，但要在当前对话中修正行为 |

**判断修正哪个 Skill**：同一个错误可能涉及多个 Skill。找到最上游的根因——如果两个候选同等可能，优先修正更通用的 Skill。

**聚类感知**：流程清晰度、失败模式编码、检查点设计是相关簇——修其中一个时，顺带扫一眼另外两个是否有同类问题。不要只盯着得分最低的维度单独修。这条来自跨 Skill 盲审实证发现（已归档至外部知识库），嵌入自修正的根因分析，不引入评分体系。

### 第二步：提议修正

当根因在工具定义时，向用户提议具体修正方案：

> 「这次错误的根因是 XXX 里 YYY 写得不够明确。建议加一条：『ZZZ』。要改吗？」

如果用户的修正建议会引入新问题，应说明理由并提出替代方案，不盲目执行。

**修正方案优先级**：先问「这个问题能不能用脚本彻底解决？」能脚本化的不靠AI判断。

| 优先级 | 方案类型 | 适用场景 | 示例 |
|--------|---------|---------|------|
| 1 | 脚本化 | 确定性逻辑：格式校验、路径判断、计数同步 | 编译后校验.sh（11项机械门控替代AI自觉）、链接有效性检查.sh（全库死链检测替代AI扫描） |
| 2 | 规则化 | 可用 checklist 或模板约束的行为 | 存档自检清单 8 项 |
| 3 | AI判断 | 无法脚本化的主观质量问题 | 内容是否是「思考记录」而非「流水账」 |

> **实证**（2026-06-29）：知识编译体系评审发现门控全凭 AI 自觉导致系统性跳过步骤。一天内创建 4 个脚本（编译收口/校验/写卡保护/链接有效性），替代了 6+ 条靠 AI 记忆的规则。脚本化的 ROI 极高——一次编写永久执行，零 AI 自律依赖。

修正类型：
- **脚本化替代**：AI判断→脚本强制执行（如「归档日计算」从规则改为脚本）
- **措辞精确化**：模糊描述→具体描述
- **补充边界条件**：只说了该怎么做→补充不该怎么做
- **新增反面案例**：只有正面示范→加反面案例
- **新增检查步骤**：缺少自检→加 checklist
- **新增铁律**：同一错误反复出现→升级为铁律
- **删除错误规则**：规则本身有问题→移除或替换
- **评审结论二次验证**：评审子代理发现的问题不一定都对——语音转写误差、归因判断、措辞偏好都可能被误判为错误。对评审结论中涉及「原文措辞」「说话人归因」的条目，在应用修正前验证是否符合常识和上下文（参考实证：2026-07-06 对话深度整理评审将「草包班→草台班子」的语音转写误差判为引用错误，AI 不加验证直接修改，导致正确内容被污染）

### 第三步：执行修正

用户确认后，按以下顺序执行：

1. 修正前用 `git diff` 或备份记录当前状态，便于回退
2. 用 Edit 工具精确修改 SKILL.md（不重写整个文件）
3. **创建新 Skill** → 必须先走 skill-creator；**修改已有 Skill** → 可直接 Edit SKILL.md
4. **改名/移动/删除前硬门控**：涉及重命名、移动、删除文件/目录/Skill 名 → 动手前必须先 `grep -r` 扫全项目引用，列出受影响文件清单，用户确认后再改。不扫就改 = 违规
5. 如果修正涉及触发机制或协作系统 → 同步检查所有调用方是否需要联动修改

### 第四步：验证 + 通知委托方

修正完成后，在当前对话中验证修正是否生效：
- 如果当前对话还能复现场景 → 按修正后的规则重新执行，确认不再犯
- 如果当前对话无法复现 → 下次执行该工具时主动留意。如果修正后同一错误再次出现，重新从第一步开始分析，考虑之前根因判断是否正确

**修复通知（硬依赖）**：如果本次自修正是由委托方触发（框架层审核或流程评估发现的问题），修复完成后在当前对话中直接通知委托方（不通过文件通知）。

**自评偏差标注**：涉及主观质量判断时（如「输出有没有 AI 味」「卡片质量好不好」），AI 自评存在偏差——同一个 AI 改的东西，同一个 AI 来评，和用户实际跑一遍的结论可能差很远。此时在验证结论末尾显式标注 `⚠️ 自评偏差风险：本验证由修正者自身执行，建议用户实际跑一次确认`。来自跨 Skill 盲审实证发现（已归档至外部知识库），轻量化为一条标注而不引入子 agent 评审。

## 主动优化流程

> v2.4 新增 · 当用户主动要求优化/审计 Skill 时触发。与纠正模式共享根因分析→提议修正→确认执行流程，区别在于触发信号是「帮我看看」而非「你做错了」。

### 触发信号

- 「优化一下 skill」「审计 skill」「帮我看看这个 skill 有没有问题」
- 「一对一优化」「逐个过一遍 skill」
- 周复盘/框架审核中发现 Skill 质量问题时 AI 主动提议

### 优化方法

与纠正模式不同，主动优化没有用户提供的具体错误信号。AI 需要自行发现问题：

1. **读全量**：通读 SKILL.md 全文
2. **对照检查清单逐项审计**（5 项）：
   - 是否有写了但做不到的触发条件或硬编码的数字/路径/版本号已过时？
   - 引用的文件/目录/Skill 是否存在、cron/hook 机制是否一致？→ 委托工具管理体检检查
   - 铁律是否可精简合并或逻辑步骤可压缩而不损失信息？
   - 是否缺反例清单（与其他 Skill 格式对齐）？
   - **是否涉及链接写入**？→ 若是，链接是否只写裸 [[链接]]（无类型前缀）
3. **提议修正**：列出发现的问题 + 建议修正方案，逐项等用户确认
4. **执行**：确认后逐项改，不批量覆盖

### 节奏控制

- 一次只列 2-4 个优化点，不堆砌
- 区分 🔴🟡🟢 优先级
- 用户说「可以」才执行，不擅自改

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 根因模糊，无法判断哪个工具 | 同一错误可能源于 CLAUDE.md 或 Skill 或 hook | 列出所有候选根因 + 各自的修正代价，让用户决定先修哪个；不因为"不确定"就不修 |
| 用户拒绝修正方案 | 用户说「不改」「不需要」 | 记录到当前对话：该纠正信号被拒绝，下次同类纠正出现时重新提议 |
| 修正后同一错误再次出现 | 已验证的修正未生效 | 重新从第一步分析——之前的根因判断可能是错的（表面修了但根因在上游） |
| 修正涉及多个工具联动 | 改 A 需要同时改 B/C/D | 一次性列出全链路改动清单，让用户看到完整影响面，逐项确认 |
| 用户纠正信号不明确 | 用户只说「不对」但没有说具体哪不对 | 先反问「具体是哪一步不符合预期？」——不确定根因不动工具 |
| 脚本化方案执行失败 | 脚本有 bug 或环境依赖缺 | 退回到规则化方案（优先级 2），暂不升级为铁律；脚本修好后再替换规则 |
| 修正目标文件被其他人同时编辑 | git status 显示冲突 | 先 stash/处理冲突，再执行修正，不覆盖他人改动 |
| 修改 Skill 自身时的保护 | 正在改 SKILL.md 时又触发新的自修正 | 🔴 排队处理：先完成当前修正→验证→再处理新的纠正信号。不同时改两个 |

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **不分析根因直接改** | 用户说「这个不对」就改当前文件，可能修的只是症状，真正的 bug 在上游 | 每次过根因分析（第一步），找最上游的修正点 |
| 2 | **自己决定怎么改不告诉用户** | 违反铁律「不自行修正」 | 必须提议→确认→执行（第二步→第三步） |
| 3 | **带着错误说「已修正」但没验证** | 违反铁律「不带着错误说完成」 | 第四步验证必须执行，不能跳过 |
| 4 | **改一处不查联动** | 违反铁律「联动检查」 | 联动检查是铁律，不是建议 |
| 5 | **对「不是工具问题」的纠正也强行改工具** | 内容问题/单次操作错误/闲聊建议被当成了工具缺陷，修改后反而引入噪音 | 严格对照「不触发的」清单逐项排除 |
| 6 | **创建新 Skill 时绕开 skill-creator 直接手写 SKILL.md** | 违反铁律「不跳过工具链」 | 创建新 Skill 走 skill-creator；修改已有 Skill 可直接 Edit |
| 7 | **把用户的轻描淡写当纠正信号** | 「以后可以试试别的方法」不是纠正，是闲聊。强行分析会导致无意义的工具修改 | 触发的四个信号必须匹配，模糊信号先反问澄清 |
| 8 | **同一个错误反复出现不升级为铁律** | 三次复现还是用规则兜底，规则太容易被忽略 | 同一错误 ≥3 次出现 → 升级为铁律 |
| 9 | **评审修正链中的假阳性传染** | 评审子代理误判正确内容为错误（如将语音转写 artifact 判为引用错误），AI 不加验证直接应用修正，导致正确内容被改错。修正后的二次错误又触发新一轮修正——形成修正震荡 | 对评审结论中涉及原文措辞、归因判断的条目，先做常识验证：同音替换大概率是转写误差、常见网络流行语优先保留原文。不确定的交用户确认后再改 |

## 铁律

- **用户纠正 = 自修正信号**：每次纠正都要过一遍根因分析，不放过任何一个可以改工具的机会
- **不自行修正**：根因分析后必须向用户提议，经确认才改。AI 不能单方面修改工具定义
- **修正要精确**：用 Edit 改具体行，不重写整个文件。改了什么、为什么改，要说清楚
- **联动检查**：改一个 Skill 时不检查关联 Skill，遗漏的副作用会在下次执行时才暴露，修复成本翻倍
- **不跳过工具链**：创建新 Skill 必须走 skill-creator；修改已有 Skill 可直接 Edit SKILL.md
- **不带着错误说完成**：带着错误说完成会让用户以为问题已解决，导致同样的错在生产中重现
- **自修正保护**：修改本 Skill 自身时，必须额外向用户确认当前修改内容，因为自修正机制的错误会影响所有后续修正
- **脚本优先**：修正方案优先用脚本/规则解决，AI判断是兜底。能用一个脚本彻底解决的问题，不要写一条规则让AI去记——AI会忘，脚本不会
- **自动执行**：纠正信号出现即自动走自修正流程（根因分析→提议修正），不需要等用户说「做自修正」。修眼前的错不算完——不接着改工具，下次还会犯
- **评审修正前先验证**：对评审子代理输出的「错误」，涉及原文措辞、说话人归因、数据精度时，先验证再改。评审结论不等于审判——评审也有盲区（语音转写 artifact、不熟悉的网络流行语、跨文件归因推断）。不确定的提交用户确认
- **内部质检不入用户输出**：Skill 模板中的自检清单、覆盖率报告、验证步骤等内部质检内容，在模板中用 `<!-- 内部自检 -->` 标注，不写入最终用户可见的产出文件。2026-07-06 实证：写作模板的自检清单被完整输出到深度整理稿末尾，用户读到一堆行号和技术标记
- **改进天花板**：同一 Skill 因同类问题已被修正 2 次但问题仍未根治时，说明问题可能不是规则层面能修的——应提议脚本化替代或结构性重构，而非继续加规则
