# 数据层体检

> 版本：v8.2 · 2026-07-14 · 维护系统全量对齐
> v8.1→v8.2：① 新增 `may_propose: [运营层巡检]`；② 新增自我进化节；③ 版本号更新

## 角色

维护系统三层之一，管知识数据层的健康。上游依赖运营层（编译脚本、调用记录同步归它），下游交接框架层（机制问题归它）、运营层（编译管线断裂归它）。被框架层审核 `may_propose` 触发。

## 路由

| 触发 | 模式 | 做什么 |
|------|------|--------|
| 「体检」「健康检查」 | **深度** | S1→S2→S3→S4，出五维报告 |
| Cron 周一、大批量编译后 | **快速** | S1 脚本扫描。全通过静默，失败升级深度 |
| 「链接审计」 | **直达链接校验** | `references/链接校验.md`，独立报告 |
| 「网络诊断」 | **直达网络诊断** | `references/网络诊断.md`，独立报告 |

> 体检检查链路：素材→编译→卡片→链接网络→MOC→遍历→AI回答。机制层问题委托「框架层审核」。

## 体检流程

### 前置判断

- `.network_index.json` >24h 未更新 → 先跑 `analyze_network.py --output-index`
- 批量编译后 → 重点看链接密度和新卡入网；周复盘后 → 重点看生命周期

---

### S1：机械扫描（快速模式）

运行 `bash AI协作工具/skills/数据层/数据层体检/S1-机械扫描.sh`，4 项脚本检查 <5 分钟。全通过 → 跳 S4。任一不通过 → 🔴 升级深度模式。

#### 1. 死链

```bash
python3 "AI协作工具/skills/数据层/数据层体检/scripts/analyze_network.py" \
  --cards-dir "02-知识卡片/知识卡片" --json
```
卡片间死链 = 0 则健康。

#### 2. 计数 + frontmatter

卡片总数 vs index.md/CLAUDE.md/人物档案三处一致。逐卡 frontmatter（created/source_type/status）和 `**来源**` 完整。

#### 3. MOC 覆盖

每卡至少被一个 MOC 收录（目标 100%）。`**来源**` 指向的素材箱文件存在（目标 ≥90%）。

#### 4. 调用记录

`00-运转日志/知识卡片调用记录.md` 卡片数与实际一致，新卡 count=1 基线存在。

---

### S2：语义与拓扑（深度模式）

> S2.1 和 S2.2 并行执行。

#### S2.1 链接审计

四步机械+语义混合检查（详见 `references/链接校验.md`）：

- **密度**：全库机械扫描，链接 <3 或 >15 的卡片列清单
- **链了不相关**：语义判断现有链接是否真有意义，预筛后逐对审计
- **应链未链**：语义相近但未互链的卡片对，启发式候选生成→审计
- **修复**：确认后补链/去链，只动 `**相关链接**` 节

#### S2.2 网络诊断

运行 `scripts/analyze_network.py --json --output-index`，检查图拓扑（详见 `references/网络诊断.md`）：

- 连通分量 = 1，关节点 = 0，平均路径 ≤ 5 跳
- Top 15 枢纽卡 + 平均聚类系数（<0.1→稀疏）
- **修复**：孤立节点补链、小分量并入主分量、关节点加冗余路径

---

### S3：生命周期（深度模式）

#### 3.1 升降级

| 升级 | 条件 |
|------|------|
| seed → growing | 调用 ≥ 3 |
| growing → evergreen | 调用 ≥ 10 |

降级：evergreen 90 天调用 ≤ 1 → growing；growing 调用 ≤ 1 → seed。

#### 3.2 滞留

seed > 60 天 → 列清单，建议补内容或删除。

#### 3.3 老化

- seed > 180 天 → `status: dormant`，MOC 保留引用标注 `<!-- dormant -->`
- 含时效性内容 > 180 天 → 标记 `⚠️ 可能过时`，列清单等用户确认
- growing > 180 天 → 检查调用记录
- evergreen > 90 天调用 ≤ 1 → 列静默清单，不擅自降级

#### 3.4 趋势

source_type 分组调用率（90 天窗口）。观察用，不作为删除依据。

## 审完怎么修

| 标记 | 含义 | 动作 |
|:--:|------|------|
| 🔴 | 死链、计数不一致、frontmatter 缺失 | 立即修复 |
| 🟡 | 补链建议、降级候选、可能过时标记 | 列清单等用户确认 |
| 🔵 | 调用率趋势、静默清单 | 记入报告，不主动修 |

| 问题类型 | 交给谁 |
|---------|-------|
| 链接密度/语义/漂移 | S2.1 子模块流程修复 |
| 网络拓扑（碎片/孤立/瓶颈） | S2.2 子模块流程修复 |
| 生命周期升降级 | 自己修（Edit frontmatter） |
| 机制层问题（脚本/配置/Skill 定义） | 委托「框架层审核」 |
| 编译管线断裂（脚本挂了、队列卡死） | 通知「运营层巡检」 |

## 铁律

- **快慢有分支**：S1 全通过 → 出报告；失败 → 走 S2+S3。不多跑
- **语义不擅自决策**：补链、合并、删除等列候选让用户裁决
- **修复必回写**：报告→修复→复扫→回写闭环
- **失败必附证据**：不通过时必须附带原始数据——卡片名、差值、点名

## 自我进化

- 同一类数据问题被检出 ≥3 次 → 不是卡片问题，是编译/链接规范问题 → 根因分析 → 向用户提议修正知识编译 Skill
- 用户纠正 → 委托工具自修正。累计 ≥3 条纠正 → 委托框架层审核深度诊断
